### 3\. Automatisierte Auswertung um 0:00 Uhr

- **Skript zur Berechnung**:
	- Ein Cron-Job (z. B. in PHP) wird ausgeführt, um alle Bewertungen des Spiels auszuwerten.
		- Durchschnittswerte für jedes Bewertungskriterium werden berechnet:
		```markdown
		Durchschnitt = SUMME(rating_value) / ANZAHL(rating_value)
		```
- **Ranking aktualisieren**:
	- Jeder Benutzer erhält Punkte basierend darauf, wie nah seine Bewertungen am Community-Durchschnitt liegen. Beispiel:
		- Community-Durchschnitt für **Teamwork**: 8,2.
				- Bewertung des Benutzers: 8.
				- Differenz: |8,2 - 8| = 0,2 (niedrigere Differenz = mehr Punkte).
- **Speicherung in der Ranking-Tabelle**:
	- Tabelle **Ranking** wird aktualisiert:
		- **total\_score** (Gesamtpunktzahl) wird basierend auf den neuen Bewertungen aktualisiert.
				- **position** (Platzierung) wird neu berechnet.

Die Bewertung müssen wir nur so weit speichern, dass sie in einem Table von einer Historie für die Gesamtbewertung, zum Beispiel über eine Saison, relevant ist. Inwieweit die einzelnen Bewertungen nach dem Spiel dann wichtig sind, oder wir die in einer Extratabelle einfacher speichern, sodass man quasi immer die eine Tabelle, die Bewertungstabelle,

Das ist eine sehr effiziente Herangehensweise. Wir könnten die **Bewertungstabelle** in zwei Teile aufteilen:

1. **Eine Hauptbewertungstabelle** für relevante, langfristig gespeicherte Daten, z. B. für Gesamtbewertungen über eine Saison.
2. **Eine temporäre oder detaillierte Bewertungstabelle**, die nur für spezifische Spielbewertungen genutzt wird und bei Bedarf in die Haupttabelle aggregiert wird.

---

### Strukturvorschlag

#### 1\. Hauptbewertungstabelle (Summary\_Ratings)

Speichert aggregierte Bewertungen pro Team oder Spieler über eine Saison oder einen Zeitraum, ohne die Details einzelner Spiele. So bleibt die Tabelle klein und performant.

| **Spaltenname** | **Beschreibung** |
| --- | --- |
| **summary\_id** | Eindeutige ID (Primärschlüssel) |
| **team\_id / player\_id** | Verweis auf das Team oder den Spieler |
| **criteria** | Bewertungsmerkmal (z. B. Teamwork) |
| **total\_value** | Gesamtsumme der Bewertungen |
| **rating\_count** | Anzahl der Bewertungen |
| **average\_rating** | Durchschnittswert |
| **season** | Saison (z. B. 2024/25) |
| **updated\_at** | Zeitpunkt der letzten Aktualisierung |

---

#### 2\. Temporäre Bewertungstabelle (Detail\_Ratings)

Speichert detaillierte Bewertungen für einzelne Spiele, die nur kurzzeitig oder für spezifische Analysen benötigt werden.

| **Spaltenname** | **Beschreibung** |
| --- | --- |
| **rating\_id** | Eindeutige ID (Primärschlüssel) |
| **user\_id** | Verweis auf den Nutzer |
| **match\_id** | Verweis auf das Spiel |
| **team\_id / player\_id** | Verweis auf Team oder Spieler |
| **criteria** | Bewertungsmerkmal |
| **rating\_value** | Bewertungswert |
| **created\_at** | Zeitpunkt der Bewertung |

---

### Datenfluss

1. **Während des Spiels**:
	- Bewertungen werden in die **Detail\_Ratings** -Tabelle eingefügt.
		- Nutzer können einzelne Kriterien bewerten (z. B. Teamwork: 8, Ausdauer: 7).
2. **Nach Spielende**:
	- Ein Skript oder Cron-Job berechnet aggregierte Werte (Summe, Durchschnitt) pro Kriterium und Team/Spieler:
		```markdown
		INSERT INTO Summary_Ratings (team_id, criteria, total_value, rating_count, average_rating, season, updated_at)
		SELECT 
		    team_id, 
		    criteria, 
		    SUM(rating_value) AS total_value, 
		    COUNT(rating_value) AS rating_count, 
		    AVG(rating_value) AS average_rating, 
		    '2024/25' AS season, 
		    NOW() AS updated_at
		FROM Detail_Ratings
		WHERE match_id = ? -- Spezifisches Spiel
		GROUP BY team_id, criteria;
		```
		- Daten werden in **Summary\_Ratings** gespeichert.
3. **Bereinigung**:
	- Die **Detail\_Ratings** -Tabelle kann nach der Aggregation bereinigt oder archiviert werden, um Platz zu sparen.
4. **Langfristige Analysen**:
	- Die **Summary\_Ratings** -Tabelle wird für Gesamtbewertungen (z. B. über die Saison) genutzt:
		- Durchschnittswerte pro Team.
				- Entwicklung bestimmter Kriterien (z. B. „Teamwork“) über mehrere Spiele.

---

### Vorteile

1. **Performance**: Die langfristige Speicherung in einer kompakten Tabelle verbessert die Abfragegeschwindigkeit.
2. **Flexibilität**: Detaildaten stehen für kurzfristige Analysen zur Verfügung, können aber entfernt werden, wenn sie nicht mehr benötigt werden.
3. **Skalierbarkeit**: Die Trennung der Daten erleichtert den Umgang mit großen Datenmengen.

### Datenbankstruktur

#### 1\. Detail\_Ratings (Bewertungen)

Diese Tabelle speichert die Bewertungen pro Spiel, Nutzer, Team oder Spieler. Anstatt jede Bewertung in einer eigenen Spalte zu speichern, wird ein JSON-Objekt genutzt.

| **Spaltenname** | **Beschreibung** |
| --- | --- |
| **rating\_id** | Eindeutige ID (Primärschlüssel) |
| **user\_id** | Verweis auf den Nutzer |
| **match\_id** | Verweis auf das Spiel |
| **team\_id / player\_id** | Verweis auf Team oder Spieler |
| **ratings\_json** | JSON-Objekt mit den Bewertungskriterien und -werten |
| **created\_at** | Zeitpunkt der Bewertung |

**Beispiel für `ratings_json`:**

```markdown
{
  "Teamwork": 8,
  "Ausdauer": 7,
  "Spielintelligenz": 9,
  "Kampfstärke": 6,
  "Siegeswille": 7
}
```

| **Spaltenname** | **Beschreibung** |
